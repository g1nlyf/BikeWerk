# AI-РОП (MVP) - Руководитель Отдела Продаж в холакратии

Статус: MVP blueprint  
Дата: 2026-02-15  
Круг: Sales Operations Circle

## 1. Миссия роли

AI-РОП отвечает за то, чтобы **каждая реальная заявка дошла до результата без потери скорости, контроля и прибыли**.

Ключевая формула роли:

- люди делают операционные действия;
- AI-РОП контролирует последовательность, сроки, полноту и эскалации;
- в кризисе AI-РОП становится командным координатором.

## 2. North Star и KPI

North Star: максимальная пропускная способность заказов при соблюдении качества и SLA.

## 2.1 Основные KPI

- `pending_manager_orders`
- median `time_to_next_status`
- доля заказов >72h без движения
- SLA breach rate по этапам
- cycle time: `pending_manager` -> `delivered`
- доля заказов без назначенного owner
- доля заказов без next action

## 2.2 Вторичные KPI

- конверсия `under_inspection` -> `awaiting_payment`
- конверсия `awaiting_payment` -> `ready_for_shipment`
- скорость реакции менеджера на новую заявку
- повторные эскалации по одному заказу

## 3. Операционный контур AI-РОП (реальная заявка)

Ниже базовый контур для вашей модели, где люди делают процесс, а AI-РОП им управляет.

1. Получена заявка -> создается order со статусом `pending_manager`.
2. AI-РОП вызывает логику назначения: выбирает наиболее разгруженного менеджера.
3. AI-РОП фиксирует `owner`, `due_at`, отправляет уведомление менеджеру в Telegram.
4. Менеджер принимает заявку.
5. Менеджер ведет коммуникацию с продавцом и выполняет 28-пунктовую проверку.
6. AI-РОП контролирует:
   - что чеклист заполнен;
   - что ключевые артефакты проверки приложены;
   - что статус переведен корректно.
7. После завершения проверки AI-РОП забирает заказ на контроль автоматически:
   - проверяет следующий шаг;
   - ставит таймер ожидания решения клиента;
   - готовит сценарии follow-up.
8. Клиент подтверждает -> AI-РОП ставит менеджеру следующий шаг:
   - согласовать отправку с продавцом;
   - зафиксировать логистику;
   - провести статус до `ready_for_shipment`.
9. На этапах `in_transit` и `delivered` AI-РОП следит за сроками и блокерами.
10. Если SLA нарушен, AI-РОП автоматически эскалирует.

## 4. Что AI-РОП контролирует постоянно

1. Очередь новых заявок и назначение owner.
2. Переходы статусов и допустимые маршруты.
3. Таймеры и SLA на каждом этапе.
4. Полноту данных перед сменой статуса.
5. Наличие следующего шага по каждому открытому заказу.
6. Наличие ответственного и дедлайна на каждом блокере.
7. Риск застревания заказа >24h/>72h.
8. Кризисные паттерны (массовая задержка, просадка конверсии, рост отмен).

## 5. Границы полномочий

## 5.1 AI-РОП может сам

- назначать менеджера на новый заказ;
- создавать задачи и SLA-таймеры;
- пересобирать приоритет очереди;
- отправлять напоминания и эскалации;
- генерировать daily/shift digest;
- предлагать action plan.

## 5.2 AI-РОП только через человека

- любые решения, влияющие на деньги клиента;
- спорные переводы статусов на риск-этапах;
- финальная фиксация компенсаций/возвратов;
- отклонение сделки по юридическим причинам (совместно с AI-GRC и человеком).

## 6. Связь с другими агентами

## 6.1 Входы для AI-РОП

- от AI-CEO: приоритеты дня, критичные сигналы.
- от AI-РОМ: качество менеджерского исполнения, персональные ограничения.
- от AI-CFO: финансовые риски по заказу.
- от AI-Head of Quality: риски по качеству сделки/проверки.
- от AI-CTO/SRE: инциденты платформы и деградации данных.

## 6.2 Выходы AI-РОП

- для AI-CEO: состояние очереди, критические блокеры, прогноз загрузки.
- для AI-РОМ: где менеджеры срывают SLA, какие навыки проседают.
- для AI-CFO: где операционная задержка ведет к утечке маржи.
- для AI-Head of Quality: сделки с повышенной вероятностью претензий.
- для AI-CTO/SRE: системные аномалии статусов/таймеров/ботов.

## 7. Кризис-менеджмент AI-РОП

AI-РОП обязан запускать playbook при следующих сигналах:

1. Spike `pending_manager_orders`.
2. >N заказов застряли >72h.
3. Массовый рост отмен на одном этапе.
4. Массовая задержка ответа менеджеров.
5. Bot/CRM отвалился и задачи перестали создаваться.
6. Неотработанные high-risk сигналы старше SLA.

Протокол кризиса:

1. Классифицировать severity.
2. Назначить owner и срок.
3. Разбить на мини-задачи по менеджерам.
4. Поднять эскалацию в админ Telegram.
5. Контролировать до закрытия и создать постмортем.

## 8. Человек + AI-РОП: рабочая модель

- Менеджер: коммуницирует, проверяет, договаривается, двигает сделку руками.
- AI-РОП: следит, чтобы менеджер не потерял шаг, срок и качество.
- Руководитель: подтверждает high-risk решения и разруливает исключения.

Модель «человек исполняет, ИИ управляет процессом» сохраняется до включения авто-коммуникации с продавцами.

## 9. Данные и события, которые нужны AI-РОП

Минимум:

- orders: `status`, `assigned_manager`, `created_at`, `updated_at`.
- order status history: `order_status_events`.
- tasks: активные задачи и дедлайны.
- manager identity and availability.
- bot delivery status (уведомление дошло/не дошло).
- CRM/Telegram action log.

## 10. Интерфейсы для MVP

## 10.1 Админка

- очередь и aging по статусам;
- heatmap SLA;
- список «горящих» заказов;
- панель кризиса: owner, SLA, next step;
- журнал решений AI-РОП.

## 10.2 Telegram

Для менеджеров:

- новый заказ;
- напоминание о SLA;
- эскалация блокера;
- чеклист «что сделать дальше».

Для админов:

- кризисный digest;
- кнопки `assign/reassign/snooze/resolve`.

## 11. Как тестировать AI-РОП

## 11.1 Unit-тесты (логика)

1. Распределение менеджеров: минимальная нагрузка выбирается корректно.
2. Правила SLA: корректный дедлайн по severity.
3. Детектор «застревания»: корректно ловит >24h/>72h.
4. Валидатор переходов статусов: запрещает нелегальные маршруты.
5. Генератор next-action: не оставляет заказ без шага.

## 11.2 Integration-тесты (backend + DB + bot)

1. Создание заявки -> назначение менеджера -> Telegram уведомление.
2. Принятие заявки менеджером -> фиксация owner и статуса.
3. Завершение 28-пунктовой проверки -> проверка полноты -> переход дальше.
4. Пропуск SLA -> автоматическая эскалация.
5. Массовая волна заявок -> корректная балансировка менеджеров.

## 11.3 E2E-тесты (бизнес-сценарии)

1. Happy path: заявка -> проверка -> оплата -> отправка -> доставка.
2. Manager delay: AI-РОП напоминает, эскалирует, перераспределяет.
3. Client undecided: AI-РОП ведет follow-up и таймер ожидания.
4. Shipment blocked: AI-РОП запускает кризисный playbook.

## 11.4 Нагрузочные и устойчивость

1. 10x входящий поток заявок за короткий интервал.
2. Деградация Telegram API.
3. Частичный отказ БД по статусным событиям.
4. Проверка, что AI-РОП не теряет owner/SLA и корректно восстанавливается.

## 12. MVP-план внедрения AI-РОП

## Week 1

1. Словарь статусов и легальных переходов.
2. SLA матрица по этапам.
3. Таблицы `ai_signals`, `ai_assignments`, `ai_sla_violations`.
4. Telegram шаблоны для manager/admin.

## Week 2

1. Авто-watchdog по SLA.
2. Кризисный список и эскалации.
3. Дашборд «очередь + горящие заказы + owner».
4. Замер KPI до/после.

## 13. Что даст MVP AI-РОП уже сейчас

- резкое снижение ручного контроля за очередями;
- прозрачность по каждому заказу «кто делает и до какого времени»;
- быстрый рост готовности к масштабированию трафика;
- фундамент для следующего шага: AI-CEO как слой стратегического арбитража.
